## Step Details

### Step-1 — Create AccountSnapshot data model
`Web/Models/AccountSnapshot.cs`

Fields: `AccountId`, `AccountName`, `AccountType`, `PreviousBalance`, `CurrentBalance`, `VarianceAmount`, `VariancePercent`, `ManualJournalCount`, `LargestManualJournalAmount`, `UnmatchedItemCount`, `DaysToCompleteReconciliation`, `HasSupportDocument`, `MaterialityThreshold`.

---

### Step-2 — Create ExceptionResult model and Severity enum
`Web/Models/ExceptionResult.cs`

Extends `AccountSnapshot`. Adds: `RiskScore` (decimal), `Severity` (enum: Low / Medium / High), `LikelyCause` (string).

---

### Step-3 — Implement DataGenerator.GenerateClean()
`Web/Services/DataGenerator.cs`

Static class. `GenerateClean()` returns `List<AccountSnapshot>` with 20 accounts across 2 periods, realistic balances with ±5–10% natural variance, all signals within normal range.

---

### Step-4 — Implement DataGenerator.GenerateDeviated() with anomalies
`Web/Services/DataGenerator.cs`

`GenerateDeviated()` returns same structure as clean but with 5–7 accounts injected with anomalies: large balance jump, missing support document, dominant manual journal, overdue reconciliation, too many unmatched items.

---

### Step-5 — Implement ExceptionScorer with weighted risk formula
`Web/Services/ExceptionScorer.cs`

Static class. `Score(List<AccountSnapshot>)` returns `List<ExceptionResult>` using:
```
overall_risk = 0.40 * variance_score
             + 0.25 * manual_journal_score
             + 0.15 * missing_support_score
             + 0.10 * unmatched_score
             + 0.10 * late_completion_score
```
Each component normalized against `MaterialityThreshold`. Maps score to Severity.

---

### Step-6 — Create DataService singleton with state and events
`Web/Services/DataService.cs`

Singleton. State: `CleanData`, `DeviatedData`, `AnalysisResult`, `LastGenerated` (enum: None/Clean/Deviated), `HasData`. Event: `StateChanged`. Methods: `GenerateClean()`, `GenerateDeviated()`, `Analyse()`.

---

### Step-7 — Register DataService in Program.cs DI container
`Web/Program.cs`

Add `builder.Services.AddSingleton<DataService>()`.

---

### Step-8 — Wire MainLayout.razor buttons to DataService
`Web/Components/Layout/MainLayout.razor`

Inject `DataService`. Add three `MudButton`s in the `MudAppBar`: Generate Random Data, Generate Deviated Data, Analyse Data (disabled when `!HasData`). Each shows a `MudProgressCircular` spinner via a local `bool _loading` flag while its operation runs.

---

### Step-9 — Implement Home.razor top panels (clean and deviated data grids)
`Web/Components/Pages/Home.razor`

Two side-by-side `MudDataGrid` panels. Columns: Account Name, Account Type, Previous Balance, Current Balance, Variance Amount, Variance %. Empty state: centered `MudText "No data yet"`.

---

### Step-10 — Implement Home.razor bottom analysis panel with state subscription
`Web/Components/Pages/Home.razor`

Bottom `MudDataGrid` with columns: Account Name, Risk Score, Severity (colored `MudChip`), Likely Cause, Variance Amount. Sorted by `RiskScore` descending. Empty state: `"Run analysis to see results"`. Subscribe to `DataService.StateChanged` in `OnInitialized`, call `StateHasChanged()` on event, unsubscribe in `Dispose()`.

---

### Step-11 — UI improvements: bold headers, reactive panels, row highlighting
`Web/Components/Pages/Home.razor`, `Web/Services/DataService.cs`, `Web/wwwroot/app.css`

Four improvements applied via TDD (11 tests):
1. **Bold table headers** — wrapped all `<MudTh>` content in `<b>` tags across all three panels.
2. **Reactive Clean/Deviated panels** — `DataService.SetDataset()` now raises `StateChanged`, so the top panels update immediately when the user clicks Generate buttons without a page reload.
3. **Contrast hover CSS** — added `.mud-table-hover tbody tr:hover:not(.selected-row)` and `tr.selected-row` styles to `app.css` for clear visual feedback.
4. **Cross-table row highlighting** — clicking an Exception Analysis row sets `_selectedAccountId`; all three `MudTable` components share a `RowClassFunc` keyed on that field, highlighting the matching account row in the Clean and Deviated panels simultaneously. Clicking the same row again deselects it.

---

### Step-12 — Add Reset Data button to clear all panels
`Web/Components/Layout/MainLayout.razor`, `Web/Services/DataService.cs`

Add a "Reset Data 🔄️" `MudButton` to the `MudAppBar` in `MainLayout.razor`. Clicking it calls `DataService.Reset()`, which clears `CleanData`, `DeviatedData`, and `AnalysisResults` back to `null` and resets `ActiveDataset` to `DataSet.Clean`. `Reset()` raises `StateChanged` so all three panels on `Home.razor` immediately revert to their empty states ("No data yet" / "Run analysis to see results").

### Step-14 — Clear analysis results when regenerating Clean or Deviated data
`Web/Services/DataService.cs`

When the user clicks **Generate Random Data** or **Generate Deviated Data**, the previously computed analysis becomes stale (it was scored against a different dataset). Update `DataService.SetDataset()` to set `AnalysisResults = null` alongside assigning the new dataset, then raise `StateChanged` once. The Exception Analysis panel on `Home.razor` already reverts to its empty state (`"Run analysis to see results"`) when `AnalysisResults` is null, so no UI changes are required — the existing state subscription re-renders the panel automatically.

Tests (bunit + xUnit):
1. After `Analyse()` then `SetDataset(Clean)` — `AnalysisResults` is null.
2. After `Analyse()` then `SetDataset(Deviated)` — `AnalysisResults` is null.
3. After `Analyse()` then `SetDataset(Clean)` — the analysis panel on `Home.razor` shows `"Run analysis to see results"`.

### Step-15 — Create AIExplanation model
`Web/Models/AIExplanation.cs`

Pure record type holding the AI-generated narrative. Fields: `Summary` (string), `DetailedExplanation` (string), `RecommendedActions` (`IReadOnlyList<string>`), `ModelName` (string), `GeneratedAt` (`DateTimeOffset`).

Tests (xUnit): property initialisation, `RecommendedActions` defaults to empty list, record equality semantics.

---

### Step-16 — Register Azure OpenAI client and bind AzureOpenAI settings
`Web/Program.cs`, `Web/Settings.cs`, `Web/appsettings.json`

Uncomment the Azure OpenAI client registration in `Program.cs`. Ensure `Settings:AzureOpenAI` binds `Endpoint`, `Key`, and `DeploymentName` into `Settings.AzureOpenAI`. Register `AzureOpenAIClient` as a singleton via DI using the bound endpoint and an `AzureKeyCredential`. No runtime behavior added yet — the client simply becomes available for injection in Step-18.

Tests (xUnit): smoke test resolving `AzureOpenAIClient` and `Settings.AzureOpenAI` from a test host when endpoint/key are configured; no network call.

---

### Step-17 — Implement ExplanationPromptBuilder (pure function)
`Web/Services/ExplanationPromptBuilder.cs`

Static class with `Build(ExceptionResult result) -> string`. Produces a structured prompt containing: account name + type, previous/current balance, variance amount + %, materiality threshold, anomaly signals (manual journal count, largest manual journal amount, unmatched items, days to reconcile, has support document yes/no), likely cause, risk score, severity. Output divided into clearly labelled sections so the LLM can return Summary / Detailed Explanation / Recommended Actions.

Tests (xUnit): snapshot-style assertion against a known `ExceptionResult`, verifying all 13 AccountSnapshot fields plus risk score and severity appear; numbers formatted `N0` / `F1`; boolean rendered as `Yes` / `No`.

---

### Step-18 — Implement IExplanationService + AzureOpenAIExplanationService
`Web/Services/IExplanationService.cs`, `Web/Services/AzureOpenAIExplanationService.cs`, `Web/Program.cs`

`IExplanationService.GenerateAsync(ExceptionResult) -> Task<AIExplanation>`. Implementation injects `AzureOpenAIClient` + `Settings.AzureOpenAI.DeploymentName`, calls `ExplanationPromptBuilder.Build`, sends a chat completion request with a system prompt instructing the model to return three labelled blocks (`Summary:` / `Detailed Explanation:` / `Recommended Actions:` — one bullet per line), parses the response into `AIExplanation`. Register `IExplanationService` as singleton in `Program.cs`.

Tests (xUnit): stubbed chat client verifies the built prompt is passed through, response parsing produces expected `AIExplanation` with Summary/DetailedExplanation/RecommendedActions populated, bullet extraction handles `-`, `*`, and numbered prefixes, and errors from the LLM surface as a clear exception type.

---

### Step-19 — Add exception detail side panel to Home.razor (click-through)
`Web/Components/Pages/Home.razor`

Reuse the existing `_selectedAccountId` (already set by `OnAnalysisRowClick`). When non-null and an analysis result exists for that account, render a `MudPaper` panel beside or below the Exception Analysis table showing: account name, type, previous / current balance, variance amount + %, all signal values, risk score, severity chip, likely cause. Include a **disabled** "Generate Explanation" `MudButton` (wired in Step-20). Clicking the same analysis row again closes the panel.

Tests (bunit): clicking a row opens the panel and renders account name + risk score + all signal values; clicking the same row closes it; selecting a different row swaps content; "Generate Explanation" button is rendered and disabled.

---

### Step-20 — Wire Generate Explanation button and display AIExplanation
`Web/Components/Pages/Home.razor`, `Web/Services/DataService.cs`

Enable the "Generate Explanation" button. On click: show a `MudProgressCircular`, call `IExplanationService.GenerateAsync(selected)`, cache the returned `AIExplanation` on `DataService` (new `IDictionary<string, AIExplanation> Explanations` keyed by `AccountId`) and raise `StateChanged`. Render Summary (bold), DetailedExplanation (paragraph), RecommendedActions (bulleted `<ul>`). A subsequent click on the same row shows the cached explanation without re-calling the LLM. `Reset()` and `SetDataset()` clear the cache so stale explanations are never shown against a new dataset.

Tests (bunit + xUnit): stubbed `IExplanationService` returns a canned `AIExplanation`; button click renders all three sections; spinner visible during the awaited call; second click on the same row reuses the cache (service called once); `Reset` and `SetDataset` clear the cache.
