## Project Overview

**CloseGuard AI Demo** is a Blazor Server MVP demonstrating AI-driven financial exception management. It imports period-over-period account balance data, scores anomalies using a weighted formula, ranks exceptions by risk, and generates controller-style narratives via Azure OpenAI.

## Commands

```bash
# Run locally (starts on http://localhost:8085)
cd Web && dotnet run

# Build solution
dotnet build CloseGuardAIDemo.slnx

# Docker build and run
docker build -f Web/Dockerfile -t closeguard-ai-demo:latest .
docker run -p 8085:8085 closeguard-ai-demo:latest
```

## Architecture

**Two-project solution** (`CloseGuardAIDemo.slnx`):
- `Library/` — shared extension methods: configuration utilities, OpenTelemetry setup (`AddOpenTelemetry()`), and default health/error endpoints (`MapDefaultEndpoints()`)
- `Web/` — ASP.NET Core 10 Blazor Server app using interactive server-side rendering

**Key dependencies**: MudBlazor 9.3 (Material Design UI), Serilog (structured logging), OpenTelemetry + Azure Monitor, Azure OpenAI (client registration is commented out in `Program.cs` — uncomment when ready)

**Config binding**: `appsettings.json` → `Settings:` section → `Settings.cs` class. Azure OpenAI endpoint/key/deployment names live under `Settings:AzureOpenAI`.

**Health endpoints** (defined in `Library/Extensions.cs`): `/livez`, `/uptime`, `/info`, `/error`

## MVP Domain Model

The core fact table is `AccountPeriodSnapshot` — one row per account per period, holding balances and anomaly signals. Exceptions are surfaced as `ExceptionCase` records with child `ExceptionSignal` rows and an `AIExplanation` generated by the LLM.

**Risk scoring formula**:
```
overall_risk = 0.40 * variance_score
             + 0.25 * manual_journal_score
             + 0.15 * missing_support_score
             + 0.10 * unmatched_score
             + 0.10 * late_completion_score
```
Each component is normalized against the account's `MaterialityThreshold`.

**AI usage pattern**: The backend calculates all signals first, then builds a structured prompt (account name, balances, variance, detected signals, likely root cause) and sends it to GPT-4.1 for a summary + detailed explanation + recommended actions. The LLM does not perform anomaly detection itself.

**Planned API surface**:
```
POST   /api/import
POST   /api/analyze
GET    /api/exceptions
GET    /api/exceptions/{id}
POST   /api/exceptions/{id}/generate-explanation
POST   /api/exceptions/{id}/feedback
```

## Development Rules

### Test-Driven Development

Always follow TDD: write a failing test first, verify it fails for the right reason, then write the minimal production code to make it pass. Never write production code before a failing test exists. Delete any production code written before its test and start over.

### GitHUb integration

When creating items in GitHub always use CloseGuardAIDemo GitHub project. Create items in Backlog and give a name to each item "Step-number short description". Add a short description to each item. Try to figure out what was the latest used step number in the current session and add +1 to each step.

When the user asks to work on a Step:
1. Create a dedicated GitHub branch named after the step (e.g. `Step-1-Create-AccountSnapshot-data-model`)
2. Move the corresponding GitHub project item to "In-progress" state
3. Do all coding on that branch

When programming is done:
1. Move the GitHub project item to "In-review" state
2. Create a Pull Request for the branch — the user will approve and merge it; never merge it yourself

When the user asks to check if a PR is approved:
1. Check PR review status with `gh pr view <number> --json reviewDecision,mergeStateStatus`
2. If approved and merged: switch to `main`, pull latest, delete the local and remote feature branch, move the GitHub project item to "Done" state, then run `/compact`
3. If not yet approved: report the current status and wait for the user to ask again
